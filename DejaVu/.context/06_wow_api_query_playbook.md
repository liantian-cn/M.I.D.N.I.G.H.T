# WoW API 查询作战手册

这份手册不是讲某个具体 API，而是讲：**遇到一个 WoW API 问题时，怎么一步步查，最后查到能下手写代码的程度。**

适合下面这些场景：

- 我记得有这个 API，但不确定 12.x 还在不在
- 我知道 API 名字，但不知道参数和返回值
- 我能查到返回结构，但不知道哪些字段可能是 `secret values`
- 我不知道该继续用旧 API，还是换成 `C_*` 新命名空间

## 总原则

### 先后顺序

1. 先用 `wow-api-mcp`
2. 再查 `warcraft.wiki.gg`
3. 还不确定，就按保守策略写

### 为什么这样查

因为这两个工具擅长的事情不一样：

- `wow-api-mcp`：查得快，适合确认 **API 叫什么、怎么调用、返回什么结构、有没有替代项**
- `warcraft.wiki.gg`：适合确认 **secret values、补丁说明、字段语义、限制条件、上下文说明**

一句话：**MCP 管“形状”，wiki 管“语义和限制”。**

## 标准查询流程

## 第 1 步：先查 API 本体

先问自己：我手里有的是什么？

### 情况 A：我知道完整 API 名

直接查，例如：

- `C_Spell.GetSpellCooldown`
- `C_UnitAuras.GetAuraDataByIndex`
- `UnitHealthPercent`

目标是确认：

- API 还存在吗
- 参数怎么传
- 返回的是普通值，还是结构体
- 有没有 wiki 链接

### 情况 B：我只知道用途，不知道名字

先按关键词搜，例如：

- `cooldown`
- `aura`
- `threat`
- `spell cast`
- `secret`

然后从搜索结果里挑最像目标的 API 再细查。

### 情况 C：我只知道旧 API 名

先查旧名字，再问两个问题：

- 这个 API 还在吗
- 它有没有新的 `C_*` 替代路径

不要直接把旧教程里的写法搬进来。

## 第 2 步：看返回值的“形状”

这一步的目标不是立刻写代码，而是搞清楚你手里拿到的东西到底是什么：

- 单个 number / string / boolean
- 多返回值
- 一个结构体
- 一个 table
- 一个对象

为什么这一步很重要？

因为 12.x 最大的坑，常常不在“函数能不能调用”，而在“返回结构里哪个字段不能直接用”。

例如：

- 冷却 API 往往返回一个结构体，不是一个简单秒数
- Aura API 往往返回很多字段，但不是每个字段都适合直接判断
- 单位相关 API 可能看起来只是 number/string，但战斗中会变成 secret

## 第 3 步：判断是不是 secret values 风险区

一旦 API 涉及下面这些内容，就要自动进入警惕状态：

- 血量、能量、怒气、法力、资源
- 冷却、充能、持续时间、剩余时间
- Aura、Buff、Debuff、层数、可驱散、可偷取
- 施法、引导、打断、目标、施法状态
- 威胁、仇恨、战斗判断
- 单位名字、单位身份、目标身份

这时不要直接相信“返回的是 number，所以就能比较”。

## 第 4 步：去 wiki 补语义

如果你已经知道 API 名字，就继续查对应 wiki 页面。重点不是整页通读，而是盯着下面几类信息：

- 参数和返回字段说明
- 补丁变更说明
- 是否提到 secret、restricted、conditional secret
- 是否提到某些字段战斗中会受限制
- 是否有替代 API 或更安全写法

如果 API 页面信息还不够，再去看：

- `Patch_12.0.0/API_changes`
- 对应补丁的 API changes 页面

## 第 5 步：把问题变成“显示”还是“判断”

这是非常重要的一步。

同一个 API 返回值，可能：

- **可以显示**
- 但 **不能拿来在 Lua 里比较和计算**

所以你一定要先把自己的需求归类：

### 只是显示

优先问：

- 能不能直接交给原生控件？
- 能不能用 StatusBar / Cooldown / FontString / Duration 之类的路径？

### 需要判断

优先问：

- 有没有 percent API？
- 有没有更粗粒度、非 secret 的状态接口？
- 有没有 `C_Secrets` 或其他官方辅助函数？

如果没有足够把握，就不要写自动判断逻辑。

## 第 6 步：选“最保守但能工作的”实现

推荐优先级：

1. 原生显示接口直接吃值
2. percent API / 非 secret API
3. `C_Secrets` 提供的辅助判断
4. 保守降级显示
5. 放弃危险逻辑

不要为了“功能看起来更聪明”去赌 secret 行为。

## 3 个常见查询范例

## 范例 1：我要做技能冷却显示

正确的查法：

1. 用 `wow-api-mcp` 查 `C_Spell.GetSpellCooldown`
2. 看返回结构有哪些字段
3. 去 wiki 看这些字段是否有 secret 风险
4. 如果只是显示，优先交给原生冷却控件
5. 不要默认自己拿 `startTime` 和 `duration` 算剩余秒数

核心结论：**冷却最容易“看着像普通数值，其实不能在 Lua 里乱算”。**

## 范例 2：我要做目标血量百分比

正确的查法：

1. 查 `UnitHealth` 和 `UnitHealthPercent`
2. 明确区分“原始值”和“百分比接口”
3. 如果只是画血条，优先让原生状态条处理
4. 如果真要百分比，优先找官方百分比 API

核心结论：**不要默认写 `UnitHealth / UnitHealthMax`。**

## 范例 3：我要判断某个 aura 是否满足条件

正确的查法：

1. 先查 `C_UnitAuras` 的具体 API
2. 看返回字段和 aura 数据结构
3. 去 wiki 看字段语义和限制
4. 区分“用于显示的字段”和“用于逻辑判断的字段”
5. 只要字段是否 secret 不明确，就先保守处理

核心结论：**Aura 是复合结构，不能把所有字段都当普通 Lua 值。**

## 什么时候必须停下来再查

只要出现下面这些情况，就不要继续凭感觉写：

- `if value then` 这里的 `value` 来自战斗相关 API
- 你准备做 `<`、`>`、`==`、`+`、`-`、`/`
- 你准备把返回值当表 key
- 你准备遍历或索引一个来源不明的 table
- 你发现旧教程和当前 wiki 说法不一致

这时候继续查，远比写完再改省时间。

## 推荐的笔记方式

以后查一个高风险 API，建议顺手记 4 件事：

- API 名字
- 返回结构
- 哪些字段可能是 secret
- 当前项目里建议怎么用

如果是高频使用的 API，可以把结论补到 `.context` 里，后面就不用重复踩坑。

## 给 DejaVu 的落地版流程

1. 先判断功能属于 `slots`、`matrix`、`panel`、`spec` 还是 `core`
2. 用 `wow-api-mcp` 查 API 形状
3. 只要涉及战斗态数据，就查 wiki
4. 优先做“能显示”版本
5. 最后才考虑是否真的需要逻辑判断

## 一句话总结

查 WoW 12.x API 时，最危险的误区不是“查不到”，而是：**查到了 API 名字，就以为自己已经理解了这个值能怎么用。**

真正安全的流程是：

- 先确认 API 形状
- 再确认返回值语义
- 再确认 secret 风险
- 最后才决定怎么写代码
