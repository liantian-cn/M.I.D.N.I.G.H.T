# WoW 12.x API 迁移笔记

## 总体判断

12.x 不是普通的小修小补，而是一次“旧经验会害人”的大换代。最要命的不是函数改名，而是：

- 一批旧 API 被废弃或替换
- 一批新命名空间被引入
- 战斗相关数据大量受 `secret values` 影响

所以迁移时不要只问“函数还在吗”，还要问“返回值还能不能像以前那样用”。

## 先查哪里

推荐顺序：

1. `wow-api-mcp`：查 API 是否存在、签名怎么变、有没有替代项
2. `warcraft.wiki.gg/wiki/Patch_12.0.0/API_changes`：看本次补丁级别变化
3. 具体 API 的 wiki 页面：看字段、限制、secret 提示

## 12.x 最重要的迁移点

### 1. Secret Values 成为默认思维模式

旧代码里最危险的部分通常不是“调用失败”，而是：

- 用返回值做比较
- 用返回值做算术
- 用返回值做布尔判断
- 用返回值决定下一步动作

这类代码都要重点复查。

### 2. 战斗统计与限制系统要换脑子

如果功能涉及战斗统计、战斗事件整理、战斗中行为判断，优先研究：

- `C_DamageMeter`
- `C_RestrictedActions`
- `C_Secrets`

不要默认还能像旧版本那样长期依赖原始战斗事件和本地推导。

### 3. 老旧全局 API 持续被 namespace API 替换

迁移时优先考虑这些方向：

- 法术：优先查 `C_Spell` / `C_SpellBook`
- Aura：优先查 `C_UnitAuras`
- 容器：优先查 `C_Container`
- 插件元数据：优先查 `C_AddOns`

只要有 `C_*` 版本，优先确认是不是已经成为主路径。

### 4. 一些 UI 旧套路不再是首选

例如：

- `UIDropDownMenu` 一类旧 UI 方案不该再当新代码默认选项
- 新配置界面优先考虑 Blizzard Settings 体系
- 旧式手搓列表、滚动、布局写法，优先看看是否已有更现代控件

### 5. TOC 版本可以更灵活

10.1 起支持逗号分隔的 `## Interface`。如果多版本代码真的完全一样，可以共用一个 TOC；如果加载文件或逻辑差异明显，还是分开更稳。

## 迁移时最值得先扫的旧代码模式

- `UnitHealth` / `UnitPower` / `UnitName` 结果直接比较或计算
- 冷却结构体字段直接参与逻辑
- 旧 Aura API 结果直接解包后做复杂判断
- 旧战斗日志路径里自己推演详细状态
- 假设 `GetSpellInfo`、旧容器 API、旧 UI 组件写法永远可用

## 新系统与新方向

12.x 文档里出现较多的新系统包括：

- `C_ActionBar`
- `C_CombatLog`
- `C_DamageMeter`
- `C_EncounterTimeline`
- `C_EncounterWarnings`
- Housing 相关命名空间

这些不一定都和本项目直接相关，但碰到对应需求时，先查新命名空间，不要先翻旧教程。

## 推荐迁移策略

### 先做“风险切片”

把代码按下面四类拆开看：

- 纯显示
- 数据读取
- 战斗判断
- 兼容旧版本

优先修“读取 + 判断”交叉区域。

### 优先改接口边界

与其到处 scattered 地打补丁，不如先把高风险 API 包成一层兼容函数。这样以后换 wiki 结论或补丁行为时，改一处就行。

### 保守胜过聪明

如果官方文档没有明确保证某个值可安全比较，就不要把它塞进复杂逻辑。

## 和本项目最相关的结论

- 本项目偏 UI / matrix / slot 组织，最容易踩的坑不是“不会调用 API”，而是“拿 secret 值自己算逻辑”
- 未来写 spec、slot、cell 时，应优先把值直接喂给 Blizzard 支持的显示接口
- 只要要做判断，就先找 percent API、非 secret 状态 API 或 `C_Secrets` 辅助接口
