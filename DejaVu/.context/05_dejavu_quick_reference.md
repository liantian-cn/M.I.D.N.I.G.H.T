# DejaVu 常见开发速查

这份是给这个项目本身用的“短路径文档”。如果你已经知道大方向，但想尽快决定代码该放哪、该用什么 API、哪些地方最容易踩 `secret values`，先看这里。

## 最常见的 6 类任务

### 1. 想加一个新的显示格子或显示行为

优先判断它属于哪层：

- 通用显示行为：放 `05_slots`
- 矩阵/单元格布局：放 `03_matrix`
- 设置项和开关：放 `04_panel`
- 某个专精专属接线：放 `06_spec`
- 通用工具或共享状态：放 `01_utils` / `02_core`

简单说：**显示放 UI 层，判断放核心层，职业差异放 spec 层。**

### 2. 想显示冷却、充能、剩余时间

先问自己：

- 我只是要显示，还是要在 Lua 里做判断？
- 这个值会不会在战斗中变成 secret？

推荐路线：

1. 先用 `wow-api-mcp` 查对应的法术 API
2. 再去 wiki 看返回结构和字段说明
3. 如果只是显示，优先交给原生冷却控件或 duration 相关能力
4. 不要默认拿 `duration`、`startTime`、`charges` 自己算逻辑

### 3. 想显示血量、能量、百分比

最稳做法：

- 显示条：优先让原生 `StatusBar` 处理
- 百分比：优先找 `UnitHealthPercent()`、`UnitPowerPercent()` 这类专门接口
- 不要在 Lua 里自己做 `UnitHealth / UnitHealthMax`

如果只是为了画条，越少自己算越安全。

### 4. 想读 aura / buff / debuff

这是高风险区。

先查：

- `C_UnitAuras`
- 对应 wiki 页面里的字段说明
- `C_Secrets` 里有没有相关辅助判断

经验法则：

- 图标、名称、显示用途和逻辑用途要分开想
- “能拿来显示”不等于“能拿来 if 判断”
- 剩余时间、层数、布尔状态都不要轻易当普通值

### 5. 想加事件监听

推荐模式：

- 在 `02_core` 集中注册和分发事件
- 业务模块只处理自己关心的更新
- 高频事件做节流，不要每次都全量刷新 UI

不要把大量业务直接塞进单个 `OnEvent` 回调里。

### 6. 想加配置项

推荐放置：

- 配置定义和面板组件：`04_panel`
- 配置默认值和读写：`02_core` 或专门的配置模块
- UI 响应配置变化：由对应模块订阅刷新

配置项不要反向把 UI、职业逻辑和存档写死在一起。

## 本项目里最容易踩坑的地方

### 冷却和充能

看起来最像“普通数字”，其实很可能是坑。只要进入战斗态，很多旧写法都不再安全。

### 单位状态

血量、能量、施法、威胁、单位名字、目标信息，都不要默认能直接判断。

### UI 先算后画

12.x 之后，很多老习惯要反过来：

- 以前：先在 Lua 里算，再传给控件
- 现在：能直接传给控件就直接传，少在 Lua 里加工

## 显示型插件的推荐做法

DejaVu 更适合优先采用这些做法：

- 百分比显示优先走官方 percent API
- 时间进度优先走 duration 对象
- 状态展示优先走颜色、亮灭、图标
- Aura 列表优先分事件更新和进度更新
- 不把多个战斗状态再拼成一个“智能分数”

## 建议的开发顺序

1. 先确认功能应该放哪个目录
2. 用 `wow-api-mcp` 查 API 形状
3. 涉及战斗态数据时，查 wiki 确认 secret 风险
4. 先写最小显示链路
5. 最后再补设置项、缓存和优化

## 常用查询模板

### 查 API 是否还存在

先查具体名字，例如：

- `C_Spell.GetSpellCooldown`
- `C_UnitAuras.GetAuraDataByIndex`
- `UnitHealthPercent`

### 查 secret 风险

组合思路：

- 先用 `wow-api-mcp` 查 API
- 再查 wiki 的 API 页面
- 再看 `Patch_12.0.0/API_changes`

### 查替代方案

遇到旧全局 API 时，优先问自己：

- 有没有 `C_*` 命名空间版本？
- 有没有 percent API？
- 有没有能直接显示的原生控件接口？

## 给 DejaVu 的一句话原则

这个项目最适合做的是：**把信息稳定地显示出来，让玩家自己判断。**

最不适合做的是：**根据战斗数据替玩家做自动决策。**
