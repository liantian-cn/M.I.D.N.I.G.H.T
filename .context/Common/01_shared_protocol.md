# MIDNIGHT 共享矩阵协议

这份文档定义 `DejaVu` 输出、`Terminal` 消费的共同显示协议。

这里只描述跨项目都必须认同的契约，不描述 Terminal 的解码实现细节，也不描述 WoW 内部原始状态。

## 角色分工

- `DejaVu` 负责把战斗信息稳定地画成一块可读取的矩阵。
- `Terminal` 负责按协议解码这块矩阵，不反向定义协议本义。
- 如果协议要改，先改这份文档，再改对应项目的专属文档和代码。

## 基本单位

- `Cell`: 4x4 像素块，主要表达布尔值或数值映射。
- `MegaCell`: 8x8 纯色块，表达明显的控制或状态信息。
- `BadgeCell`: 8x8 图标块，右下角带 2x2 脚标。
- `CharCell`: 8x8 文本像素块，用来表达简单字符或数值。

坐标统一按 `Cell` 计算，不按像素计算。

## 当前运行契约

- 当前实现里的矩阵尺寸是 `84 x 28` 个 `Cell`。
- 这是一条当前仓库依赖的运行契约，不应被写死成“永远不变的真理”。
- 如果矩阵尺寸要改，必须同步更新协议文档、DejaVu 输出侧和 Terminal 解码侧。

## 区域与锚点语义

- 协议同时包含动态内容区和稳定定位区。
- `MARK_POINT` 这一类近黑色标记点属于稳定锚点语义，用于定位矩阵区域。
- 颜色、脚标类型、区域分区都属于协议字段，不只是视觉效果。
- 分区装饰色和业务数据色要分开看，不能把纯装饰区域当业务锚点。

## 协议边界

- 协议描述的是“屏幕上要长成什么样”，不是 WoW 内部 API 直接透出。
- Terminal 可以在自己的专属文档里补充帧校验、解码输出结构、线程链路，但这些内容不应反向改写协议定义。
- 任何跨项目依赖的结构变化，都先落回这份文档。
